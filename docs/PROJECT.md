## Architecture

### Routing & Locale Model

Astro i18n is configured in `astro.config.mjs`:

- Default locale: `es` (served at `/`, no prefix)
- Secondary locales: `en` (`/en/...`), `zh` (`/zh/...`)
- `prefixDefaultLocale: false` — Spanish has no URL prefix

### i18n Pattern

All user-visible strings live in `src/i18n/{es,en,zh}.ts` as flat key-value TypeScript objects. The `t()` helper in `utils.ts` resolves a key to the current locale's value. **Every new user-visible string added to any component MUST be added to all three locale files** (`es.ts`, `en.ts`, `zh.ts`) simultaneously.

### Component Pattern

- One Astro component per major page section (named after the section)
- Props: always include `lang: 'es' | 'en' | 'zh'` and `t: (key: string) => string`
- Styling: scoped `<style>` blocks inside each component, referencing CSS custom properties (design tokens)
- Client scripts: inline `<script>` tags at the bottom of the component, wrapped in an IIFE

### Testimonials Pattern

Testimonials are maintained in:
1. `src/i18n/{es,en,zh}.ts` — keys: `testimonials.N.name`, `testimonials.N.location`, `testimonials.N.text` (and optionally `testimonials.N.summitDate`)
2. `src/components/Testimonials.astro` — a `testimonials` array with one object per entry, each referencing `t('testimonials.N.*')` and containing hardcoded `initials` (2-letter abbreviation) and `photo` (Firebase Storage URL)
3. Rendering: `testimonials.map()` auto-generates carousel slides and navigation dots — no manual wiring needed for new entries

Testimonial numbering is non-contiguous: testimonials 1, 2, 3, 5, 6, 7, 8, 9, 10, 11 exist; testimonial 4 was intentionally skipped.

Photo assets are hosted on Firebase Storage (`firebasestorage.googleapis.com/v0/b/aconcagua-co.firebasestorage.app/o/...`).

### Deployment

The site deploys to Firebase Hosting (`firebase.json` present). Build output goes to `dist/` via `astro build`.
